# inbox 패턴 미적용 결정

## 상황

과제 C는 알림 발송 시스템 구현을 요구하지만, outbox/inbox 패턴 사용 여부를 직접 명시하지는 않는다.

비동기 처리, 실패 재시도, 서버 재시작 후 미처리 재처리, 다중 인스턴스 중복 방지 요구사항을 만족하려면 발송 작업을 DB에 남기는 outbox는 필요하다고 판단했다.

반면 inbox는 별도로 도입할지 판단이 필요했다. 특히 "동일 이벤트에 대해 알림이 중복 발송되면 안 된다", "동시에 같은 요청이 여러 번 들어오는 경우도 고려하라"는 요구사항을 멱등성으로 풀어야 하는데, 이 멱등성 책임을 어디에 둘지가 쟁점이었다.

## 해석

inbox 패턴의 핵심은 멱등성 자체가 아니다.

더 본질적인 목적은 외부에서 들어온 원본 요청이나 이벤트를 우리 시스템 안에 먼저 저장하고, 이후의 처리 책임을 내부로 가져오는 것이다.

예를 들어 결제사의 webhook, 배송 상태 변경 webhook, 다른 서비스에서 발생한 수강 신청 완료 이벤트처럼 외부 시스템이 보내는 원본 이벤트는 언제든 중복 전달되거나 재전송될 수 있다.

이런 이벤트를 inbox에 먼저 저장하면, 이후의 해석, 가공, 재시도, 중복 판단을 우리 시스템 안에서 통제할 수 있다. 멱등성은 이 구조에서 따라오는 효과에 가깝다.

## 외부 원본 이벤트 vs 정리된 알림 요청

inbox 도입 여부는 우리가 받는 데이터의 성격에 따라 달라진다.

외부에서 원본 이벤트를 그대로 받는다면 inbox가 의미 있다.

예를 들면 다음과 같은 데이터다.

- 수강 신청이 완료되었다
- 결제가 승인되었다
- 배송 상태가 변경되었다

이 경우 우리 시스템은 원본 이벤트를 해석해서 누구에게 어떤 알림을 보낼지, 어떤 채널로 보낼지, 어떤 메시지를 만들지 결정해야 한다. 이 해석 과정 자체가 실패하거나 재시도 대상이 될 수 있으므로, 원본 이벤트를 inbox에 먼저 저장하는 구조가 자연스럽다.

반면 이미 알림 발송에 필요한 정보가 정리된 요청이라면 별도 inbox 레이어의 필요성이 낮다.

예를 들면 다음 값들이 이미 들어오는 경우다.

- 수신자 ID
- 알림 타입
- 참조 데이터
- 발송 채널

이 값들이 이미 결정되어 있다면, 요청을 받은 시점에 바로 알림 요청 내용을 저장하고, 나중에 발송해야 할 작업도 함께 저장할 수 있다.

## 근거

이 과제는 후자에 해당한다고 해석했다.

알림 발송 요청 API는 이미 알림을 만들기 위해 필요한 정보를 거의 다 받은 상태라고 보았다.

요청 안에는 수신자 ID, 알림 타입, 참조 데이터, 발송 채널이 들어온다. 즉 서버 입장에서는 요청을 받은 순간 이미 다음 내용을 알 수 있다.

- 누구에게 보낼 알림인지
- 어떤 종류의 알림인지
- 어떤 사건이나 데이터와 관련된 알림인지
- 이메일로 보낼지, 인앱 알림으로 보낼지

따라서 서버가 원본 이벤트를 다시 분석해서 알림 대상으로 바꾸는 단계가 필요하지 않다.

예를 들어 "수강 신청이 완료되었다"라는 원본 사건만 들어온다면, 서버는 그 사건을 보고 누구에게 어떤 알림을 보낼지 따로 판단해야 한다. 이런 경우에는 먼저 원본 사건을 저장해두고, 나중에 알림 요청으로 변환하는 구조가 의미 있다.

하지만 이 과제의 API는 이미 "이 사용자에게, 이 타입의 알림을, 이 채널로 보내 달라"는 형태에 가깝다. 그래서 요청을 받은 시점에 바로 알림 요청 내용을 저장하고, 동시에 나중에 발송해야 할 작업도 함께 저장할 수 있다.

이렇게 하면 요청을 받은 사실은 DB에 남고, 실제 발송은 별도 처리 흐름에서 진행된다. 외부 발송이 실패하더라도 이미 저장된 알림 요청 자체가 사라지지는 않는다.

결국 이 구조에서는 요청을 한 번 더 별도 저장소에 넣어두고, 다시 알림 요청으로 변환하는 단계의 필요성이 크지 않다고 판단했다.

또한 과제 범위에는 원본 이벤트를 받을 별도 진입점이 없다. 메시지 브로커는 사용하지 않고, 결제 webhook이나 메일 발송 서비스의 callback 같은 외부 수신 경로도 요구사항에 포함되어 있지 않다.

## 결과

inbox는 도입하지 않는다.

본 구현에서는 알림 요청 API가 이미 정리된 알림 발송 요청을 받는다고 해석하고, 요청을 받은 시점에 알림 요청 내용과 발송해야 할 작업을 하나의 흐름으로 저장한다.

알림 요청 내용은 사용자가 조회할 상태와 목록의 기준이 된다. 발송해야 할 작업은 API 요청 스레드와 분리되어 별도 처리 흐름에서 실행되며, 실패 시 재시도 횟수와 실패 사유를 남긴다.

따라서 현재 요구사항에서 별도 inbox 테이블은 책임이 중복될 수 있어 오버엔지니어링이라고 판단했다.

다음 조건이 추가되면 inbox 도입을 재검토한다.

- 외부 원본 이벤트, webhook payload, broker message를 그대로 수신하는 경로가 추가되는 경우
- 원본 이벤트를 알림 타입, 수신자, 채널로 해석하는 과정도 실패/재시도 대상으로 관리해야 하는 경우
- 원본 이벤트 자체를 감사 로그로 보관하거나 재처리해야 하는 경우
